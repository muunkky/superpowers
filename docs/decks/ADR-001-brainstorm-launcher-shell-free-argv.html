Architecture Decision

ADR-001 · brainstorming companion launcher · Accepted 2026-07-11

We run the brainstorm launcher shell-free: a quote-aware argv tokenizer plus execFile, no shell.

Replacing cp.exec(BRAINSTORM_OPEN_CMD + url) with an in-tree tokenizer and cp.execFile closes the shell-injection class a security audit flagged as blocking adoption — the honest cost is that shell-only conveniences stop working, which is exactly the point.

ADR-001 · uncommitted · made with gitban MCP

In plain terms

The launcher runs an operator's command through a shell — the classic shell-injection shape.

The brainstorming companion can auto-open a browser using a command an operator supplies (BRAINSTORM_OPEN_CMD). Today it hands that command to a shell, which interprets whatever it contains.

Instead, we split the operator's command into a program plus its arguments ourselves — honoring quotes, so a quoted path with spaces stays one argument — and run the program directly with no shell, exactly like the built-in launcher two lines below already does.

The tradeoff we accept: shell-only conveniences (pipes, $VAR, ~, backslash-escaping) stop working — and that is precisely the point, because the shell layer we remove is the vulnerability.

Why now

A security audit classified the shell sink High — "Not usable as-is" — and it gated an org's adoption.

server.cjs (maybeOpenBrowser(), ~line 540) hands BRAINSTORM_OPEN_CMD concatenated with the companion URL to a shell via cp.exec(cmd + url). An AI/skill-based security audit — upstream obra/superpowers issue #1957 — flagged this shape and blocked that org from adopting the plugin.

  • The path's four pre-launch guards — opt-in BRAINSTORM_OPEN, loopback-only bind, skip-if-connected, open-at-most-once — mitigate but do not remove the structural injection shape.
  • A static/structural scan re-fires on the cp.exec(string) sink regardless of the guards or any prose about the trust model.
  • Superpowers is zero-dependency: no shell-parsing library may be added.

The safe pattern already lives one branch away: at server.cjs:547 the built-in platform launcher already does cp.execFile(bin, args) with the URL as a discrete argv element. The fix makes the override branch match it.

The decision

Replace the shell exec with an in-tree, quote-aware argv tokenizer feeding execFile — no shell.

Before — the flagged sink:

cp.exec(BRAINSTORM_OPEN_CMD + url)
// operator string + URL → /bin/sh -c

After — structurally identical to the built-in branch:

const argv = parseLauncherCommand(cmd);
cp.execFile(argv[0], [...argv.slice(1), url]);
  • parseLauncherCommand splits on whitespace outside quotes and keeps single/double-quoted spans intact (stripping the quotes) — ~16 lines of standard-library JavaScript, exported for unit testing, and it never throws.
  • The call site guards with if (argv.length && argv[0]): a whitespace-only, empty, or quoted-empty (['']) value falls through to the built-in platform launcher rather than reaching execFile('', …).
  • The four pre-launch guards and the built-in branch are untouched.

Why it works

With no shell, an injected ; rm -rf ~ becomes an inert argv token — not a command.

This removes the flagged sink structurally — with no shell, the audit's structural match can no longer fire. Injected shell metacharacters, whether in the operator value or the URL, are inert literal tokens: the injection class is closed, not merely mitigated by the guards.

The tradeoff, stated honestly

Shell-only features drop — and that dropped shell layer is itself the security property.

Preserved

  • A binary, its flags, and quoted argument forms.
  • All quoted spaced paths — "/opt/My Browser/bin" stays one argument.
  • The existing lifecycle test's JSON-quoted form (node "<abs>" "<abs>") — the tokenizer strips the quotes to the exact argv the capture script needs.

Dropped (by design)

  • Pipelines, redirection, command substitution, logical/sequencing operators, globbing, $VAR expansion.
  • ~ (tilde/home) expansion — ~/bin/browser now ENOENTs.
  • Backslash-escaping — /opt/My\ Browser/bin no longer joins.

Only two forms that worked under the shell regress, and each has a one-time, operator-visible migration: rewrite a backslash-escaped or tilde-prefixed path as a quoted absolute path.

This is acceptable — indeed intended — because the dropped shell layer is the security property: an injected ; rm -rf ~ becomes an inert literal argv token handed to a browser binary, never a command. The PRD's Non-Goals decline shell pipelines; its Future Considerations gate re-adding any shell path on evidence of real use.

Alternatives considered

Three other mechanisms were weighed in DD-001 and rejected — recorded here by reference, not re-argued.

(a) Quote-aware tokenizer + execFile — chosen

Removes the sink structurally and preserves realistic launcher use (binary, flags, quoted spaced paths). Reads as the execFile pattern already in the next branch.

(b) Naive whitespace split + execFile

Smallest code, but retains the literal quote characters — so it breaks the lifecycle test's JSON-quoted paths and splits quoted spaced paths. Fails "tests stay green" and "realistic use." Rejected.

(c) Documentation-only trust-model note

Zero code, but leaves the cp.exec(string) sink in place, so a structural scan re-fires regardless of prose — betting the outcome on the reviewer overriding their own tooling. Its doc deliverable is kept and folded into (a). Rejected as the primary fix.

(d) Keep the shell string + add a pre-split array env var

Preserves full shell capability, but leaves the flagged sink on the default variable and adds a second config surface for a capability the PRD declines to commit to. Rejected.

Weighed in full in DD-001 against the PRD's "solved"; the ADR records the conclusion, not the deliberation.

Consequences

The injection class closes; a minor operator migration is the honest cost.

Positive

  • The structurally-flagged cp.exec(string) sink is gone; an equivalent audit no longer matches the shell shape on the override path. Both launch branches now use execFile with the URL as a discrete argv element.
  • Injected shell metacharacters — in the operator value or the URL — are inert literal tokens: the injection class is closed, not merely mitigated by the guards.
  • BRAINSTORM_OPEN_CMD gains operator-facing documentation (shape + trust posture + explicit no-shell note) it entirely lacked.

Negative (honest)

  • Shell-only capability through BRAINSTORM_OPEN_CMD is lost. Operators using backslash-escaped or ~-prefixed paths face a minor, visible migration; anyone relying on pipes/redirection/substitution loses it. Documented, not silent.
  • The tokenizer is a deliberately minimal quote handler (whitespace + single/double quotes only); it does not emulate every shell quoting edge case, by design.

Neutral

  • One new exported pure function and a one-branch call-site edit in a single file — no new modules, call paths, state, schema, or external surface, and no new dependency. Rollback is a single-commit revert.

How it's validated

A required end-to-end no-execution test proves the sink is dead — and it is what catches a future revert.

The spawn test in tests/brainstorm-server/lifecycle.test.js launches the server with a malicious BRAINSTORM_OPEN_CMD = node "<capture>" "<marker>" ; touch "<pwned>" and asserts:

  • (a) <pwned> never appears on disk — no shell interpreted the ; touch; and
  • (b) the override still fired once — the capture ran.

It deliberately does not assert URL delivery: post-fix the injection tokenizes to [node, capture, marker, ';', 'touch', pwned] and execFile appends the URL last, so the capture's argument for the marker is ';', not the URL. URL delivery is locked separately by the existing clean "auto-opens once" test and the tokenizer unit assertions.

This spawn test — not the tokenizer unit tests — is what catches a future revert of the call site to cp.exec(argv.join(' ')), which would keep every unit test green while silently reopening injection.

  • Unit assertions on parseLauncherCommand (metacharacters inert, quoted spaced path stays one arg, lifecycle shape strips quotes, empty / quoted-empty / unmatched-quote edges).
  • Both existing lifecycle override tests pass with no assertion weakened; the full tests/brainstorm-server suite is green on Linux; the diff adds no dependency.

Accepted — and what it rests on

Accepted and in force: the shell-free form stands unless real operators prove they need the shell.

Necessary but not sufficient. Clearing the structural shell shape removes the single confirmed finding (#1957) — but the reporting org's adoption also rests on the PRD-scope assumption that #1957 is the whole product problem. If the org has undisclosed audit findings, this change clears the scan without unblocking adoption (see DD-001's residual-risk row).

Revisit trigger. Evidence that real operators depend on shell features through BRAINSTORM_OPEN_CMD — the migration proving too costly in practice — reopens a deliberate, documented shell path per the PRD's Future Considerations. Absent such evidence, the shell-free form stands.
