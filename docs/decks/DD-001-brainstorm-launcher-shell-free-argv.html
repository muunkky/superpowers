DD-001 · brainstorming companion · Draft

Retire the browser-launcher's shell shape by tokenizing the operator command into argv and launching with no shell

A security audit flagged a shell-injection shape in the brainstorming companion's auto-open launcher. This design converges on the smallest change that removes it: a ~16-line in-tree quote-aware tokenizer feeding cp.execFile, making the flagged branch structurally identical to the shell-free launcher two lines below it.

DD-001 · uncommitted · 1/16 · made with gitban MCP

The problem

One override branch hands an operator string to a shell — the classic injection shape, and an auditor blocked adoption on it

The companion server auto-opens a browser the first time a screen is ready. When an operator sets BRAINSTORM_OPEN_CMD, that branch concatenates the operator's value with the companion URL and hands it to a shell via cp.exec.

What "shell-injection shape" means, plainly: because a real shell parses the string, characters like ;, |, or $(…) in that string are read as commands, not as text. Anything that can influence the value can run its own program.

An org's security audit — upstream obra/superpowers issue #1957 — classified this shape High / blocking and gated the org's adoption of the plugin on it being gone or made explicitly trusted.

The sharp irony: two lines below the finding, the built-in platform-launcher branch already does the shell-free thing — cp.execFile(bin, args) with the URL as a discrete argument. The fix is to make the flagged branch match the one right next to it.

Current state · the sink

Four guards protect the launch; then one branch goes through a shell and its neighbor does not

Grounded in skills/brainstorming/scripts/server.cjs, maybeOpenBrowser(). The four pre-launch guards are sound and stay untouched; the asymmetry sits only in the last two branches:

if (!process.env.BRAINSTORM_OPEN) return;                 // opt-in guard
if (HOST !== '127.0.0.1' && HOST !== 'localhost') return;  // loopback-only guard
if (clients.size > 0) return;                             // skip-if-connected guard
const url = companionUrl();                               // carries the session key

if (process.env.BRAINSTORM_OPEN_CMD) {
  cp.exec(process.env.BRAINSTORM_OPEN_CMD + ' ' + JSON.stringify(url), () => {});
  return;                        // <-- the finding: cp.exec invokes a shell
}
const launcher = browserLauncherForPlatform(url);
cp.execFile(launcher.bin, launcher.args, () => {});   // no shell — the pattern to match

browserOpened = true is set at the top, so the once-only guard holds regardless of which branch runs. module.exports already exports internal functions for testing, so adding one more export is idiomatic, not a new pattern.

Definition of done

"Solved" is an outcome — an equivalent audit no longer flags the branch — with five invariants held intact

From the PRD, "solved" is stated as an outcome, not a mechanism: an equivalent audit no longer classifies the override path as High/blocking, while all five of these hold:

  • The override still launches a realistic launcher — a binary + flags + quoted/spaced path args, the shape the existing tests exercise.
  • The four pre-launch guards are untouched.
  • The built-in launcher path is behaviorally unchanged.
  • The existing tests/brainstorm-server suite stays green.
  • No new dependency is added.

Plus one addition: BRAINSTORM_OPEN_CMD gains the operator-facing documentation it entirely lacks today. Any candidate is judged against this bar — not against "is the code smaller."

DD-001 · uncommitted · 4/16 · made with gitban MCP

The approach · candidate (a)

Parse the override into argv with a small quote-aware tokenizer, then launch via execFile — no shell anywhere

Chosen: candidate (a) — a hand-rolled quote-aware tokenizer feeding cp.execFile, making the override branch structurally identical to the built-in branch the audit already accepts. The key tradeoff is visible and deliberate: full shell power is traded for a shell that is simply not there.

~16lines of tokenizer (standard-library JavaScript, no dependency)
~8lines changed at the call site

The whole change — tokenizer + call site + one export — is ~24 lines in one file. No new modules, no new call paths, no change to how maybeOpenBrowser is reached. The tokenizer earns its keep precisely because honoring quotes is what preserves realistic launcher use (quoted paths with spaces, flags) — the exact capability a naive split would lose.

DD-001 · uncommitted · 5/16 · made with gitban MCP

The exact edit

The override branch swaps cp.exec(string) for parseLauncherCommand + execFile — the URL becomes a discrete argv element

Before — the flagged sink:

cp.exec(
  process.env.BRAINSTORM_OPEN_CMD
    + ' ' + JSON.stringify(url),
  () => {}
);
return;

After — argv + execFile, no shell:

const argv = parseLauncherCommand(
  process.env.BRAINSTORM_OPEN_CMD);
if (argv.length && argv[0]) {
  cp.execFile(argv[0],
    [...argv.slice(1), url], () => {});
  return;
}
// empty / quoted-empty / whitespace-only:
// fall through to the built-in launcher

The argv[0] conjunct in the guard is deliberate: a quoted-empty value ("") tokenizes to [''] — length 1, empty binary — and the guard prevents an execFile('', …). The return moves inside the guard so a blank or quoted-blank override falls through to the platform default rather than silently doing nothing.

Alternatives weighed

The PRD handed four candidates; three fail the "solved" bar, and the fourth reads as consistent with the code beside it

(b) Naive whitespace split + execFile

Smallest code, removes the shell — but breaks the existing lifecycle test (retains quote characters) and splits quoted spaced paths. Rejected.

(c) Documentation-only trust model

Zero code change. But the flagged cp.exec sink remains, so a structural scan re-fires regardless of prose. Its doc deliverable is kept and folded in. Rejected as the fix.

(d) Keep shell string + add an array env var

Preserves shell power — but the sink stays on the default variable, and it adds a second config surface for a capability the PRD declines to commit to. Rejected.

(a) Quote-aware argv tokenizer + execFile

Removes the sink entirely, preserves realistic launcher use, keeps the lifecycle test green, and reads as consistent with the execFile pattern two lines below. Chosen.

DD-001 · uncommitted · 7/16 · made with gitban MCP

Why the tempting shortcuts fail

The naive split breaks the test's JSON-quoted paths; docs-only never removes the structural sink

Why (b) fails — concretely. The existing lifecycle test's openCaptureCommand produces a value with literal double-quotes: node "<abs>" "<abs>". Splitting on whitespace yields:

['node', '"<abs>"', '"<abs>"']   // quote characters RETAINED

So node is asked to open a file literally named "…", the capture never fires, and the test goes red. It also fails requirement 4 — a quoted spaced path splits into two arguments. The quote-aware tokenizer instead strips the quotes: ['node', '<abs>', '<abs>'] — exactly what the test needs.

Why (c) fails. An auditor could accept the shape as documented — but "solved" must clear the audit's class of check: a static/skill-based scan that flags the shape. Prose does not remove the cp.exec(string) sink; the structural scan re-fires regardless. It bets the whole outcome on one reviewer overriding their own tooling.

DD-001 · uncommitted · 8/16 · made with gitban MCP

The key insight · parity, not narrowing

Honoring quotes is parity with the old shell — no quoted launcher form regresses; only shell-only features and tilde/backslash drop

The subtle point that must not be mistaken for a regression: under the old cp.exec path, the shell already word-split the value, so a spaced binary path already required the operator to embed quotes in the value ('"/Applications/My Browser.app/…"'). The tokenizer reproduces exactly this — it splits on whitespace outside quotes and keeps quoted spans intact.

Preserved — every quoted launcher form

Binary + flags + quoted spaced-path arguments all pass through identically. The tested lifecycle shape and requirement 4's <binary> "<path with a space>" both hold.

Dropped, deliberately — two shell-quoting forms

Backslash-escaped spaces: /opt/My\ Browser/bin tokenizes to two args (backslash-escaping not honored). Tilde paths: ~/bin/browser passes literally and execFile ENOENTs. Workaround — quote the absolute path — is in the docs.

This is why (a) is a faithful, near-parity replacement — every quoted launcher form is preserved — rather than a silent narrowing.

DD-001 · uncommitted · 9/16 · made with gitban MCP

The security property

Dropping shell features is not a cost of the fix — it is the fix: an injected metacharacter becomes an inert string

execFile spawns the binary directly, so pipes, redirection, command substitution, &&/||/;, globbing, $VAR, ~, and backslash-escaping are not honored — their characters survive as inert literal argv tokens. That inertness is the security property itself.

Risks & failure modes

The residual risks are all low-likelihood, and each is met by docs or the best-effort nature of auto-open

RiskImpactLikelihoodMitigation
An operator depends on shell features and their launch silently stops working. Medium Low Docs state the no-shell contract; auto-open is best-effort and the URL is always shared as a fallback, so a failed override never blocks the session.
Tokenizer diverges from shell word-splitting on some quoting edge case. Medium Low Contract is deliberately minimal (whitespace + single/double quotes, matching the launcher forms tests and built-ins use); unit tests cover the lifecycle shape, spaced quoted path, unmatched quote, empty input.
The org's auditor still flags something after the shell is gone. High Low The finding is explicitly the shell shape (#1957); removing cp.exec eliminates the structural match. Undisclosed findings are an upstream PRD-scope question, not a design defect.
Whitespace-only fall-through masks a genuine operator typo. Low Low Documented; the alternative (no-op on blank) is strictly worse. Truly empty strings already fell through under the old code, so this only extends existing behavior to whitespace.
DD-001 · uncommitted · 11/16 · made with gitban MCP

The regression lock

Unit tests alone would stay green through a revert to the shell — only an end-to-end no-execution test locks the property

Testing is two tiers, both required. The unit tier asserts parseLauncherCommand treats metacharacters as inert tokens (open ; touch pwned → ['open', ';', 'touch', 'pwned']), keeps quoted spaced paths as one arg, and strips the lifecycle-shape quotes.

But the unit tier is not enough. A future revert of the call site to cp.exec(argv.join(' ')) would keep every unit test green while reopening injection — the tokenizer never special-cased ;, so its tests never notice the sink came back. Only asserting non-execution catches that.

Required end-to-end test (in lifecycle.test.js). Spawn the real server with a malicious override — node "<capture>" "<marker>" ; touch "<pwned>" — and assert:

  • (a) <pwned> never appears on disk — the ; touch was interpreted by no shell;
  • (b) the override fired exactly once — the capture ran, the marker file exists and is non-empty.

It intentionally does not re-assert URL delivery: post-fix the URL is appended after the inert ;, so it is no longer at the position the capture script reads. URL delivery stays separately locked by the existing clean "auto-opens once" test and the tokenizer units. This test's sole job is proving non-execution of the sink.

Implementation shape · single phase

One vertical slice, built test-first, touching four files and adding no dependency

The whole change is one phase — vertically complete. Built in this order:

  1. Write the failing tests first

    Unit assertions (metacharacter-inert, quoted-spaced-path, lifecycle-shape, edge cases incl. quoted-empty → ['']) in browser-launcher.test.js, plus the required no-execution spawn test in lifecycle.test.js.

  2. Add parseLauncherCommand and export it

    ~16 lines of standard-library JavaScript in server.cjs; total and best-effort — never throws, an unmatched quote closes at end-of-string.

  3. Rewrite the override branch to argv + execFile

    The call-site swap, with the argv.length && argv[0] guard and whitespace-only fall-through; in-code comment updated to "no shell — argv via execFile."

  4. Document BRAINSTORM_OPEN_CMD

    A compact block in visual-companion.md: accepted shape, opt-in/loopback trust posture, and the explicit "no shell — these features are not honored" note. Quote spaced paths; do not backslash-escape or use ~.

Rollback is one revert — the change is confined to server.cjs, the two test files, and visual-companion.md. No state, no schema, no external surface.
