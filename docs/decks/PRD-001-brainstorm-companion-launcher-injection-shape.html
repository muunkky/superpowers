PRD-001 · obra/superpowers #1957 · Draft

Retire the shell-injection shape that a security audit says blocks adoption of Superpowers

An organization wants to run the Superpowers plugin. Their security audit flagged one thing and stopped them: the companion opens the browser by handing an assembled command to a shell — the classic shell-injection shape. Nothing malicious is happening, but that shape alone is enough for an audit to block adoption.

PRD-001 · uncommitted · 1/12 · made with gitban MCP

The problem

A security audit rated the companion "Not usable as-is (High)" — and that verdict blocks a real org from adopting the plugin

Superpowers' brainstorming skill ships a small local "visual companion": a loopback web server that auto-opens a browser tab the first time there's a screen to show. It is part of the plugin, so any org adopting the plugin inherits it.

The reporter of upstream issue #1957 wants their organization to allow Superpowers. Their org runs a skill-based, AI-driven security audit as an adoption gate. That audit returned a blocking verdict:

"Not usable as-is (High)." Until the flagged signal is gone — or the trust model is made explicit enough that an auditor signs off — orgs that gate adoption on a clean security scan cannot use the plugin.

The person hurt is the operator trying to bring the plugin into a security-conscious org. The block is not hypothetical: it is a filed verdict standing between them and adoption.

What the audit flagged

The single confirmed finding: the operator override builds a command by string-concatenation and hands it to a shell

The companion has two mutually-exclusive ways to open a browser. Only one was flagged — and the shape of the finding is the asymmetry between them.

In plain terms: when an operator sets the BRAINSTORM_OPEN_CMD environment variable, the server glues that value together with the page's URL into one text string and asks a system shell to interpret it. Handing an assembled string to a shell is the classic "shell-injection shape" — and static / AI audits flag that shape on sight, regardless of the guards around it.

The telling detail

The shell-free pattern the audit is asking for already lives two lines below the finding

The built-in platform launcher (the non-override branch) already avoids the shell entirely: it picks open / rundll32.exe / xdg-open per platform and passes the URL as a discrete argument via execFile. Its own in-code comment says it does this "so a url-host containing shell metacharacters can't inject a command."

The finding — override branch

cp.exec(
  process.env.BRAINSTORM_OPEN_CMD
    + ' ' + JSON.stringify(url),
  () => {}
)  // cp.exec invokes a shell

The reference — built-in branch, ~2 lines below

cp.execFile(
  launcher.bin,
  launcher.args,   // url is one argv element
  () => {}
)  // no shell

So the repo already contains the exact safe idiom the finding wants — just not on the override path. That asymmetry is the entire finding, and it means a fix can match a pattern already in the file rather than import a new philosophy.

Why we believe it — and the honest caveats

The audit conceded a reviewer could accept it as shipped, yet recommended hardening — and we reasoned this from the issue and the code, not our own audit

Two things travel with this into the pull request:

The audit was not absolute. It acknowledged the mitigating context — the path is opt-in, loopback-only, and the value is operator-supplied — and said a reviewer could accept it as shipped. It recommended hardening anyway. So this is a "should fix," not an unarguable exploit.

Grounded in a filed issue. This addresses a real reported problem — issue #1957, raised from an org's own security audit — developed by reasoning from that issue plus direct inspection of server.cjs. The full authoring-environment disclosure travels in the PR, per obra/superpowers' contribution rules.

A further gap widens the problem surface: BRAINSTORM_OPEN_CMD has no operator-facing documentation. It appears only in server.cjs comments and in one test file — nowhere an operator or auditor would look. So an operator has nothing telling them how to use the override safely, and an auditor has no user-facing statement of the intended trust model to weigh against the flagged shape.

Definition of success

"Solved" is an audit outcome — deliberately not a specific mechanism

The definition the room needs to agree on:

An equivalent security audit or reviewer re-examining the operator-launcher path — the class of audit that produced #1957 — no longer classifies it as High-severity or as blocking.

Crucially, how that outcome is reached is left open on purpose. Whether the shape is retired by parsing the command shell-free, by documenting the trust model so the auditor accepts it, or by offering a shell-free alternative alongside today's behavior is an engineering decision — routed downstream, not pre-decided here. The audit outcome, plus the invariants on the next slide, is the yardstick every candidate is measured against.

This keeps the PRD in its lane: it states the problem and the bar for "done," and stays out of the way of the design that will choose the mechanism.

What "solved" must not break

Whatever the fix, five invariants of today's behavior must survive it

The audit outcome only counts as "solved" if all of these continue to hold:

  1. The override still launches real commands. An operator's chosen browser command still works for its realistic use — a binary, optional flags, and path arguments that may be quoted or contain spaces.
  2. The four pre-launch guards are unchanged: opt-in via BRAINSTORM_OPEN, loopback-only bind, skip when a client is already connected, and open at most once.
  3. The built-in platform launcher is behaviorally unchanged — it is already shell-free and stays that way.
  4. The existing tests/brainstorm-server suite stays green — with no assertion weakened. New tests may be added.
  5. No new dependency is introduced — Superpowers is zero-dependency by design.

Do not silently regress the override. The override is intentionally a full command so an operator can point it at a real launcher, including quoted paths with spaces and flags. Narrowing that capability without a deliberate, recorded decision is a regression, not a fix.

The committed surface

Four capabilities are committed — and every one must hold for any mechanism the design lands on

None of these presupposes a particular parsing strategy; they are the acceptance surface a fix is graded against.

CapabilityAcceptance
Clear the audit finding Re-running the audit's class of check (or a skeptical security reviewer reading the code) no longer classifies the BRAINSTORM_OPEN_CMD path as High / blocking — the concern is eliminated, or the trust posture is made explicit enough that the reviewer accepts the path as shipped.
Preserve the override for realistic use A binary-plus-arguments command still launches the companion once, with the URL delivered as a usable, key-carrying argument (the existing lifecycle test passes unchanged). A quoted path containing a space reaches the process as one argument, not split.
Preserve all surrounding behavior Gating is unchanged (opt-in, loopback-only, skip-if-connected, once-only — the guard tests stay green). The built-in launcher branch is behaviorally identical: still shell-free, still URL-as-argv.
Document the override A user-facing doc states that BRAINSTORM_OPEN_CMD exists, the command shape it accepts, that it is opt-in and loopback-only, and the trust model it runs under — so an operator and an auditor can find the contract without reading server.cjs.

Scope boundary

This is exactly one finding — and everything the audit praised stays untouched

The same audit explicitly praised the rest of the server: the auth token, timing-safe comparison, Origin check, CSP, path-traversal guards, and chmod 600. This PRD authorizes work on the operator-launcher path only; it is not a mandate for general "security hardening."

In scope

  • The BRAINSTORM_OPEN_CMD branch of maybeOpenBrowser.
  • User-facing documentation of the variable.
  • Test additions that lock the chosen behavior (without weakening existing assertions).

Out of scope — and why

  • Every other security surface — praised, not flagged; touching it is scope creep.
  • The built-in launcher branch — already safe; it stays the reference pattern, not edit surface.
  • The pre-launch guards — correct; the finding is about how it launches, not whether.
  • Any new dependency — the zero-dependency constraint is hard.

Why the tight line matters. This ships to a repo with a ~94% PR rejection rate that closes speculative fixes and scope creep on sight. The change must be crisp, grounded in the real finding, and consistent with the surrounding safe pattern.

The open decision — routed to the ADR

One thing is deliberately unresolved: how much override capability to keep, and by what mechanism

This PRD does not choose. The design doc weighs these candidates against the "solved" yardstick; the ADR records the decision.

(a) Quote-aware argv parsing, no shell

Retires the shell shape and preserves quoted-path / flag launchers. Requires in-tree quote handling (no dependency) and a call on how far to honor edge cases like open -a 'Google Chrome'. The fork's current leaning — recorded as an input, not a mandate.

(b) Naive whitespace split, no shell

Simplest way to drop the shell — but regresses quoted paths with spaces and breaks the existing lifecycle test as written. Likely disqualified by "solved"; recorded for completeness.

(c) Documentation-only trust-model clarification

Leaves the code as-is and makes the opt-in / loopback / operator-trust posture explicit so an auditor accepts it. Lowest code risk — but this org's auditor may still flag the shape regardless of documented context.

(d) Keep the shell path, add a pre-split array form

Preserves full shell capability while offering a shell-free alternative. But it leaves the flagged shape present on the string form, so it may not clear the audit on its own.

The marked panel is the leaning, not a decision — the ADR owns the call.

Risks & assumptions

The sharpest risk: an org's tooling may flag the shape mechanically, so only removing the shell clears it

Product risks

  • A doc-only or partial fix may not be accepted. If the org's tooling flags the shape mechanically, only options that remove the shell clear it — a factor the design must weigh against the audit's stated flexibility.
  • Over-narrowing the override. Choosing a mechanism that drops a launcher form some operator relies on trades one problem for another; the "realistic-use" bound is the guard.

Load-bearing assumptions (each could be wrong)

  • This one finding is the whole problem. If adoption hinges on other undisclosed findings, clearing the scan won't unblock the org.
  • "Realistic use" is a launcher invocation — a binary plus flags plus quoted paths — not arbitrary shell pipelines. If some operator relies on pipes or substitution, a shell-free approach would regress them.
  • Green lifecycle tests = correctness. They set the override to a double-quoted command; a fix that broke them would break the behavior contract they encode.

A ground-truth constraint for the design. A naive whitespace split would keep the literal quote characters in the argv and break the existing lifecycle test — the design must account for that when weighing options.
